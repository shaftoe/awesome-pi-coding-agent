1. The Big Picture

Data flows through a 3-stage pipeline: discover → enrich → generate. The first two stages determine what ends up on disk. The third just renders it.

Key insight: A raw search result from an API goes through 5 checkpoints before becoming a stored entry — early filter, full filter, dedup, classify, and save. Most results are discarded at the first two. 
  ┌─────────────┐     ┌──────────────┐     ┌──────────────┐
  │  DISCOVER   │     │    ENRICH    │     │   GENERATE   │
  │             │     │              │     │              │
  │  npm ───────┤     │ GitHub meta  │     │  README.md   │
  │  GitHub ────┤────▶│ README scan  │────▶│              │
  │  YouTube ───┤     │ Classify     │     │              │
  │  pi-deps ───┤     │ Health score │     │              │
  └──────┬──────┘     └──────────────┘     └──────────────┘
         │
         ▼
  Filter → Dedup → Save to data/

2. Step 0: Where Candidates Come From

Four discoverers run in parallel. Each iterates a set of search queries against a different API.

🟦 npm Registry

source: "npm-search"

Queries: keyword (keywords:pi-package) + broad text (pi-agent)

ID: full npm package name, e.g. @scope/pi-extension

URL: https://npmjs.com/package/...

Smart pagination: keyword queries fetch all pages; broad queries stop early when relevance drops.

🟪 GitHub Search

source: "github-search"

Queries: text ("pi-coding-agent") + topics (topic:pi-extension)

ID: owner-repo (slash → hyphen)

URL: https://github.com/owner/repo

Pre-enriched: the search API already returns stars, forks, topics, archived, etc.

🟥 YouTube Data API

source: "youtube-search"

Queries: exact phrases ("pi coding agent") + broad with exclusions

ID: YT_<videoId>

URL: https://youtube.com/watch?v=...

Requires YOUTUBE_API_KEY. Paginated, with fatal abort on 403.

🟩 pi-mono Deps

source: "github-search"

Queries: GitHub code search for @mariozechner/pi-coding-agent in package.json

ID: owner-repo

Catches repos that depend on pi-mono packages but don't mention "pi" in their name/description.

Early rejection: Inside each discoverer, every raw API result passes through earlyReject() — a fast O(1) check that drops blacklisted URLs, blocked scopes (@stdlib/*), and blocked names ("pi") before creating a full candidate object. This eliminates ~60% of results immediately.

3. Step-by-Step Flow Diagram

What happens to a single candidate as it moves through saveNewCandidates() in src/index.ts:

🔍 Candidate arrives from discoverer

{ url, source, hint?, id?, metadata? } — a DiscoveryCandidate

Example: { url: "https://npmjs.com/package/@scope/pi-mcp", source: "npm-search", id: "@scope/pi-mcp", metadata: { description: "...", github_url: "..." } }

passed earlyReject() inside discoverer

🛡️ Full Relevance Filter — isRelevant()

Runs the FULL_RULES pipeline — 15 composable FilterRule objects evaluated in priority order.

Layer 1 (hard blocks): blacklist → blocked scope/name → Raspberry Pi → math π → PixiJS → Pi Network → industrial → ecosystem → forks → OpenAPI → non-Latin script → non-English Latin

Layer 2 (positive signals): name patterns → text signals → topics → keywords

Layer 3 (default): YouTube = reject, others = accept

Rejections are auto-blacklisted — the URL is added to data/blacklist.json so future runs skip it instantly.

verdict.accept === true

🔗 3-Axis Dedup — checkDuplicate()

Checks the candidate against 3 indices built from all existing entries:

① URL: primary identity — exact match on entry.url

② ID: canonical identifier — e.g. @scope/name or owner-repo

③ GitHub URL: cross-source — GitHub discoveries checked against metadata.github_url in npm entries

All 3 use O(1) Map lookups. If any axis matches → duplicate, skip.

isDuplicate === false

💾 Build Entry → Classify → Save

1. Build Entry object: Derive id from URL (or use candidate's explicit id). Extract name from metadata. Set initial health to { score: 0, level: "stale" }.

2. Classify: classifyEntry() runs a rule chain: YouTube URL → video, text keywords → category, README scores → best match, fallback → tool.

3. Save: Write to data/{category}s/{id}.json. Scoped packages encoded as @scope--name.json.

4. Update indices: The new entry is added to the dedup maps so subsequent candidates in the same run are checked against it.

entry saved to disk, enrichment runs next

🔬 Enrichment Pipeline — runEnrichment()

For each non-enriched entry (batched 5 at a time, 200ms delay):

① GitHub metadata: Resolve metadata.github_url → fetch stars, forks, license, last_commit, language, topics, archived, fork

② README analysis: Fetch raw README → count category keywords → _readme_category_scores

③ Re-classify: Run classifier again with richer data

④ Health score: calculateHealth() → score 0–100 → level (active ≥70 / maintained ≥40 / stale ≥15 / dead <15)

⑤ Re-save: If category changed, move file to correct folder. Overwrite JSON.

enriched entry on disk

📝 Generate README — generateReadme()

Read-only: loads all entries, groups by category, sorts by health score, renders markdown tables.

Output: README.md — does not modify any data files.

4. The Filter: How We Decide What Stays

The filter is implemented as a chain of 15 composable rules. Each rule returns a verdict: { accept: true }, { accept: false, reason }, or null (pass to next). First non-null verdict wins.

Two Pipelines

PipelineRulesRuns WherePurpose
EARLY_RULES 3 rules blacklist, blocked-scope, blocked-name Inside each discoverer O(1) fast rejection before candidate creation
FULL_RULES 15 rules early 3 + 8 more negative + 4 positive + 1 default saveNewCandidates() Complete 3-layer relevance check

All 15 Rules in Priority Order

L1 blacklist

O(1) Set lookup. Skips known-bad URLs.

L1 blocked-scope

@stdlib, @aws-sdk, @pixi, etc.

L1 blocked-name

Exact name set: pi, micromark, etc.

L1 raspberry-pi

"raspberry", "rp2040", "gpio" in text/topics

L1 math-pi

"digits of pi", "irrational number" in text

L1 pixijs

"pixi" in name/URL/topics

L1 pi-network

"pi network", "pi cryptocurrency" in text

L1 industrial

"aveva", "osisoft", "pi server" in text

L1 ecosystem

Tiptap, SAP, Pimcore, Node-RED, Storybook…

L1 fork

"oh-my-pi" non-compatible fork signal

L1 openapi

"openapi specification", "swagger-ui"

L1 non-latin

CJK, Cyrillic, Arabic, Devanagari script

L1 non-english-latin

Stop-word detection: German, French, Indonesian… (YouTube only)

L2 positive-name

pi-*, @scope/pi-*, pi-mcp patterns

L2 positive-text

"pi coding agent", "pi.dev", "pi extension"

L2 positive-topics

pi-agent, pi-coding-agent, pi

L2 positive-keywords

pi-package, pi-extension, pi-theme

L3 youtube-strict

YouTube: require positive signal. Others: accept.

Extending the filter: To add a new rule, define a FilterRule object and insert it into FULL_RULES at the desired position. If it's O(1), also add it to EARLY_RULES. No other code changes needed.

6. The Enrichment: Where Metadata Comes From

Enrichment runs after all candidates are saved. It processes entries that haven't been enriched yet (tracked via sentinel fields in metadata).

Enrichment Steps per Entry

① GitHub Metadata

Resolves metadata.github_url (for npm entries) or the primary URL (for GitHub entries). Fetches /repos/{owner}/{repo} API. Injects: stars, forks, open_issues, language, license, last_commit, topics, is_archived, is_fork, homepage.

② README Analysis

Fetches raw README from raw.githubusercontent.com (tries main then master). Counts keyword occurrences per category → _readme_category_scores.

③ Re-classify

Runs classifyEntry() again with the richer data (especially README scores). If category changes, moves the JSON file to the correct folder.

④ Health Score

calculateHealth() uses GitHub signals (stars, recency, archived, fork) or YouTube signals (views, likes, age). Score 0–100 mapped to active/maintained/stale/dead.

⑤ Save

Overwrites the JSON file with enriched data. Category may have changed the folder.

Health Scoring Formula

SignalPointsApplies To
Base+50All entries
Stars / 10+0 to +20 (capped at 200 stars)GitHub entries
Commit < 30 days ago+20GitHub entries
Commit 30–90 days ago+10GitHub entries
Commit > 365 days ago−20GitHub entries
Archived repo−50GitHub entries
Fork−30GitHub entries
Views / 500+0 to +15YouTube entries
Likes / 50+0 to +10YouTube entries
Published < 30 days ago+25YouTube entries
Published > 365 days ago−20YouTube entries

Final score clamped to 0–100. Levels: active ≥70 maintained ≥40 stale ≥15 dead <15

7. The Storage: What Lives on Disk

Each entry is a single JSON file. The folder = the category. The filename = the ID (with encoding).

Directory Structure

│   ├── pi-mcp.json
│   ├── pi-simple-memory.json
│   └── @scope--pi-tool.json    ← @scope/name → @scope--name.json
├── tools/               ← category = "tool"
│   └── owner-repo.json         ← GitHub: owner/repo → owner-repo.json
├── providers/
├── themes/
├── templates/

8. Worked Example: One Package's Journey

Follow @0xkobold/pi-cursor from discovery to disk:

① npm discoverer runs query "pi-agent"

npm Search API returns this package. earlyReject() checks: not blacklisted ✓, not @stdlib ✓, not blocked name ✓ → passes.

② Candidate created

{ url: "https://npmjs.com/package/@0xkobold/pi-cursor", source: "npm-search", id: "@0xkobold/pi-cursor", metadata: { description: "pi extension for Cursor AI...", keywords: ["pi-package"] } }

③ isRelevant() runs FULL_RULES

Passes blacklist (not listed) → passes blocked scope (@0xkobold not blocked) → passes blocked name → no Raspberry Pi signals → no math pi → ... → positive-keywords rule matches "pi-package" → { accept: true }

④ checkDuplicate()

Not in URL map, not in ID map, no GitHub URL to check → isDuplicate: false

⑤ Build + classify + save

id = "@0xkobold/pi-cursor", name from metadata, initial health { 0, "stale" }. Classifier sees "pi extension" → extension. Saved to data/extensions/@0xkobold--pi-cursor.json

⑥ Enrichment: GitHub metadata

Finds metadata.github_url = "https://github.com/0xkobold/pi-cursor" → fetches /repos/0xkobold/pi-cursor → injects stars=0, last_commit, language="TypeScript", etc.

⑦ Enrichment: README analysis

Fetches README → counts keywords → { extension: 10, tool: 7, skill: 1 }

⑧ Re-classify + health

README scores confirm "extension" (highest score). Health: 50 base + 0 stars + 20 (commit < 30d) = 70 → active. Re-saved.

⑨ README.md generation

9. The Funnel: Numbers at Each Stage

Typical pipeline run — approximate numbers based on the current data:

🔍 Raw API results from 4 discoverers ~18,000 +
↓ earlyReject() inside discoverers
📦 Candidates returned ~4,500
↓ isRelevant() full filter + auto-blacklist
✅ Pass relevance filter ~3,000
↓ checkDuplicate() 3-axis dedup
🆕 New unique entries ~50–200 per run
↓ classifyEntry() + save to disk
💾 Written to data/*.json ~50–200
↓ runEnrichment() + enrichVideos()
🔬 Fully enriched entries 2,662 total on disk
Why so few new entries per run? The pipeline runs daily. After the first run ingested ~2,600 entries, subsequent runs mostly find already-known packages. New entries come from recently published packages or newly tagged repos.
